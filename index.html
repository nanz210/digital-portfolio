Project 1

Account Services Harmonised Dashboard

This project focused on creating a standardised dashboard view for Account Services, allowing senior management to compare KPIs across different operations departments more easily.

Power BI Hue SQL Data Preparation Dashboard UI

Problem: Different departments had dashboards that were built in different formats, making it difficult for senior management to compare KPIs clearly.

Project Goal: To create a harmonised dashboard layout with consistent visuals, KPI displays, and reporting structure across different operations departments.

My Role: I helped create the base dashboard UI that other dashboards would follow, while also working on the data side for the Monitoring View and the full development of the People View.

User Communication: I communicated with users to better understand what they needed from the dashboard and how the layout could be designed to support their work.

Outcome: The dashboard helped standardise reporting across different operations departments, making it easier to view, compare, and understand key operational metrics.

Dashboard Functionality

Monitoring Overview

The Monitoring View tracks operational KPIs such as received cases, open cases, completed cases, SLA status, aging cases, pending documents, and product details.

People Overview

The People View tracks staff availability, overtime above 2 hours, odd-hour work patterns, and 7-day staff trends.

KPI Cards

The KPI cards show staff availability by team, such as how many staff are available out of the total team size, together with the percentage shown visually.

Interactive Filtering

The blue and light-blue proportion bars on the KPI cards also work as filter buttons, allowing users to view specific team records in the work details section.

Trend Charts

The line charts allow users to filter staff availability, overtime above 2 hours, and odd-hour trends by clicking on specific data points.

Detailed Tables

The dashboard includes detailed tables for product details, SLA summary, aging summary, and work pattern details.

Technical Work and Challenges

Connecting to Real Data

For the Monitoring View, I mainly worked on the data side by connecting the Power BI dashboard to a real table in Hue. This required checking whether the existing visuals and fields still worked after replacing dummy data with real data.

Understanding Data Fields

One challenge was understanding what each real data column meant. Unlike dummy data, where column names are usually clear, real data sometimes used less obvious field names. For example, a column name may not directly say “product” even if it represents product information.

Handling Complex Tables

The real table had around 66 columns and many distinct values. To build the dashboard properly, I had to understand the data structure before mapping the correct fields to each visual.

Building Dynamic Measures

For the People View, many KPI cards required multiple measures. Since the dashboard originally used static placeholder values, these had to be rebuilt into dynamic measures once the real data was available.

Measure Dependencies

Some measures depended on other measures, so changing one calculation could affect multiple visuals. This required careful rebuilding, testing, and replotting to make sure the dashboard worked correctly.

Scale of Work

The People View required more than 50 measures because each KPI card needed several calculations to display the correct values, percentages, and filtering behaviour.

Project 2

Asset Management Dashboard

This project focused on preparing and transforming operational data into a dashboard-ready dataset for tracking asset management records, SLA status, ownership information, and reporting timelines.

Python Pandas Power BI Power Query DAX ETL Pipeline

Project Purpose: To transform multiple operational data sources into a clean, standardised, and dashboard-ready reporting dataset.

Problem: The source files had different structures, fields, and reporting requirements, making it difficult to use them directly in Power BI.

My Role: I worked on the data preparation and transformation process by creating Python scripts, preparing lookup tables, standardising datasets, and supporting Power BI reporting logic.

Dashboard Use: The dashboard helps users monitor asset management records, SLA timelines, overdue records, planned items, completed items, and detailed breakdowns by ownership or reporting hierarchy.

Outcome: The final output was a standardised reporting dataset that could be used in Power BI to support clearer tracking, filtering, and SLA reporting.

Scripts Created

Script 1: Lookup Table Creation

This script created a reference lookup table by extracting unique values from a master dataset. The lookup was used to map a primary business attribute to its corresponding reporting hierarchy.

Script 2: Regional Mapping

This script generated a reference table for geographical ownership information. It applied both primary mapping and fallback logic to improve coverage where values were missing.

Script 3: Main ETL Pipeline

This script loaded two operational datasets, standardised their structures, combined them, enriched them using lookup tables, normalised metric values, created SLA bands, and exported the final reporting file.

Technical Work

Standardising Multiple Sources

The two source files had different column structures, so I had to reshape them into a common format before combining them into one reporting dataset.

Building Lookup Logic

I created lookup tables to map classification fields, reporting hierarchy, ownership groups, and regional information. These mappings helped enrich the final dataset.

Cleaning and Transforming Data

I cleaned key fields, removed records that could not be evaluated, and converted inconsistent metric values into standard numeric values.

Creating SLA Bands

I derived SLA and aging categories so the dashboard could display whether records fell within different timeline bands, such as short, medium, long, or overdue periods.

Power Query Transformations

In Power BI, I also worked with Power Query steps such as changing data types, replacing values, adding custom columns, and unpivoting selected columns to make the data easier to visualise.

DAX Measure Logic

I worked with DAX measures to calculate source counts and support SLA matrix visuals. This helped the dashboard display values based on the selected timeline or aging bucket.

Reflection

Project Reflections

This section reflects on my learning experience from the dashboarding and data engineering projects I worked on during my internship.

Harmonised Account Services Dashboard Reflection

What Went Well

I was able to contribute to a dashboard that helped standardise reporting across different operations departments. Creating a harmonised layout made the dashboard easier for users and senior management to understand and compare.

I also improved my Power BI and data preparation skills by working on the Monitoring View and People View. For the People View, I gained more experience creating dynamic measures and building KPI cards that could display availability, percentages, and filtering behaviour.

What Did Not Go Well

One challenge was moving from dummy data to real data. Dummy data was easier to understand, but the real Hue table had many columns, unclear field names, and different values that needed to be interpreted correctly before they could be used in Power BI.

Another difficulty was replacing static placeholder values with dynamic measures. Some measures depended on other measures, so one change could affect multiple KPI cards and visuals.

What Can Be Improved

I can improve by spending more time understanding the real data structure before building or updating visuals. This would help me map the correct fields faster and reduce the need to rework visuals later.

I can also improve by documenting measure logic more clearly, especially when measures have dependencies. This would make troubleshooting easier and help future users understand how the dashboard calculations work.

Summary

Overall, working on the Asset Management Dashboard helped me understand that dashboard development is not only about making visuals look consistent. It also requires understanding user needs, preparing reliable data, building accurate measures, and testing how each part of the dashboard works together.

Asset Management Dashboard Reflection

What Went Well

I was able to improve a manual and time-consuming data preparation process. Initially, the data had to be mapped manually, which took more than 5 hours because around 2,000 rows had to be checked and mapped.

By using Python to create mapping files and automate part of the ETL process, I reduced the work to about 30 minutes. This made the refresh process much more efficient, especially because the data originally had to be refreshed every week.

What Did Not Go Well

The original manual mapping process was very inefficient and not practical for a workplace setting. Repeating a 5-hour task every week would take up too much time and increase the chances of human error.

Halfway through the project, the structure of the incoming data changed, which caused the pipeline to break. This showed me that even an automated process can fail if the input data changes unexpectedly.

What Can Be Improved

I can improve the pipeline by adding stronger validation checks before the script runs fully. For example, the script could check whether the expected columns exist, whether row counts look correct, and whether key fields are missing or renamed.

I can also improve by documenting the mapping logic and refresh steps more clearly, so that the process is easier to maintain and troubleshoot in the future.

Summary

Overall, this project helped me understand the importance of automation, efficiency, and data validation. By moving from manual mapping to a Python-based ETL process, I was able to save time and create a more centralised workflow. When the data structure changed, tracking the number of rows during each refresh helped me identify the issue quickly, and having the logic centralised in Python allowed me to fix the problem in one place.
